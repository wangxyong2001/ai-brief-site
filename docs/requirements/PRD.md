## 1. 业务背景与目标

### 1.1 背景

AI 领域发展迅猛，每日产生大量论文、产品更新和开源项目。从业者面临信息过载问题，难以高效获取关键动态。现有解决方案要么覆盖不全，要么缺乏智能筛选。

### 1.2 目标

构建自动化 AI 每日简报系统，实现：
- **聚合**：整合 arXiv、OpenAI、Anthropic、GitHub 等核心数据源
- **去重**：基于语义相似度智能去重，避免重复内容
- **分发**：生成结构化简报，支持多渠道推送

### 1.3 核心价值

| 维度 | 价值 |
|------|------|
| 效率 | 用户每日 5 分钟掌握核心动态 |
| 质量 | 语义去重 + 优先级排序，过滤噪音 |
| 覆盖 | 技术论文 + 产品发布 + 开源项目 |

## 2. 用户角色与场景

### 2.1 目标用户

| 角色 | 特征 | 核心诉求 |
|------|------|----------|
| AI 研究员 | 学术机构/企业研究院 | 追踪最新论文、研究趋势 |
| AI 工程师 | 企业 AI 团队 | 了解工具更新、开源项目 |
| 产品经理 | AI 产品公司 | 竞品动态、市场趋势 |
| 技术决策者 | CTO/技术总监 | 行业动向、战略参考 |

### 2.2 核心场景

**场景 1：每日早间速览**
> 用户打开简报页面，5 分钟内浏览昨日 AI 领域核心动态，包括重要论文、产品发布和热门开源项目。

**场景 2：深度追踪**
> 用户点击感兴趣的内容，查看原文详情，支持按来源、时间、类别筛选。

**场景 3：历史检索**
> 用户通过关键词搜索历史简报，回顾特定主题的发展脉络。

## 3. 功能需求清单

### 3.1 优先级定义

- **P0**: MVP 必须，无此功能无法上线
- **P1**: 重要功能，显著提升用户体验
- **P2**: 增强功能，锦上添花
- **P3**: 未来迭代

### 3.2 功能列表

| ID | 功能 | 描述 | 优先级 | 验收标准 |
|----|------|------|--------|----------|
| F01 | RSS 数据采集 | 定时拉取 arXiv/OpenAI/Anthropic RSS | P0 | 每日准时采集，失败有告警 |
| F02 | GitHub Release 采集 | 监控指定仓库的 Release 动态 | P0 | 新 Release 24h 内收录 |
| F03 | 语义去重 | 使用 GLM Embedding API 计算向量相似度去重 | P0 | 相似度 > 0.85 视为重复 |
| F04 | 向量存储 | LanceDB 存储内容向量 | P0 | 支持向量检索，查询 < 100ms |
| F05 | 元数据存储 | SQLite 存储标题、链接、时间等元数据 | P0 | 数据持久化，支持 CRUD |
| F06 | 简报生成 | 按日期聚合内容，生成结构化简报 | P0 | 每日自动生成 |
| F07 | Web 界面 | FastAPI 后端 + 前端展示 | P0 | 响应式设计，移动端友好 |
| F08 | 内容分类 | 自动识别论文/产品/开源项目等类别 | P1 | 准确率 > 85% |
| F09 | 重要性评分 | 基于热度、来源、影响力评分排序 | P1 | Top 内容优先展示 |
| F10 | 全文搜索 | 支持关键词搜索历史内容 | P1 | 搜索响应 < 500ms |
| F11 | 订阅推送 | 支持 Email/RSS 订阅 | P2 | 支持退订，频率可控 |
| F12 | 用户偏好 | 自定义关注领域、数据源 | P2 | 个性化简报 |
| F13 | API 接口 | 提供 REST API 供第三方集成 | P2 | 有文档，有鉴权 |
| F14 | 数据导出 | 导出 JSON/CSV 格式历史数据 | P3 | 支持按时间范围导出 |

### 3.3 数据源配置

| 来源 | 类型 | 采集频率 | 内容类型 |
|------|------|----------|----------|
| arXiv cs.AI/cs.CL/cs.LG | RSS | 每 6 小时 | 学术论文 |
| OpenAI Blog | RSS | 每 6 小时 | 产品/研究 |
| Anthropic Blog | RSS | 每 6 小时 | 产品/研究 |

## 4. 非功能需求

### 4.1 性能要求

| 指标 | 目标值 |
|------|--------|
| 页面加载时间 | < 2s (P95) |
| API 响应时间 | < 500ms (P95) |
| 向量检索延迟 | < 100ms |
| 并发用户数 | 支持 100 并发 |

### 4.2 可用性要求

| 指标 | 目标值 |
|------|--------|
| 系统可用性 | 99.5% |
| 数据采集成功率 | 99% |
| 故障恢复时间 | < 30min |

### 4.3 安全要求

- HTTPS 强制加密
- API 接口需鉴权（JWT）
- 敏感配置（API Key）使用环境变量
- 数据库定期备份

### 4.4 可维护性

- 结构化日志（JSON 格式）
- 健康检查端点 `/health`
- 采集失败告警通知
- 代码覆盖率 > 70%

## 5. 数据需求

### 5.1 数据模型

**Item（内容条目）**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| title | TEXT | 标题 |
| summary | TEXT | 摘要 |
| source | TEXT | 来源（arxiv/openai/anthropic/github） |
| source_url | TEXT | 原文链接 |
| category | TEXT | 分类 |
| published_at | DATETIME | 发布时间 |
| collected_at | DATETIME | 采集时间 |
| importance_score | FLOAT | 重要性评分 |
| embedding_id | TEXT | 向量 ID（关联 LanceDB） |

**DailyBrief（每日简报）**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 主键 |
| date | DATE | 简报日期 |
| item_ids | JSON | 包含的条目 ID 列表 |
| created_at | DATETIME | 生成时间 |

### 5.2 数据保留

- 原始内容：保留 90 天
- 向量数据：保留 90 天
- 简报记录：永久保留

### 5.3 数据量估算

| 维度 | 估算值 |
|------|--------|
| 每日新增条目 | 50-200 条 |
| 去重后条目 | 30-100 条 |
| 每条向量大小 | ~2KB |
| 月数据增量 | ~50MB |

## 6. 约束条件

### 6.1 技术约束

| 约束 | 说明 |
|------|------|
| 后端框架 | FastAPI（Python 3.11+） |
| 向量数据库 | LanceDB（嵌入式，无服务器依赖） |
| 元数据库 | SQLite（轻量级，易于部署） |
| Embedding | GLM Embedding API（智谱 AI） |
| 部署域名 | ai.tomabc.com |

### 6.2 资源约束

- 单机部署，无分布式架构
- 预算有限，优先使用免费/开源方案
- 开发周期：2 周 MVP

### 6.3 合规约束

- 遵守各数据源的使用条款
- 保留原文链接，不存储全文内容
- 用户数据最小化收集
